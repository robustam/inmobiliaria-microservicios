1. ¿Qué es una arquitectura de microservicios?

Una arquitectura de microservicios divide una aplicación grande en servicios pequeños e independientes, donde cada uno tiene una responsabilidad específica, su propia base de datos y se comunica con los demás a través de HTTP (APIs REST).

Flujo general del sistema:

Cliente (Postman / App) → API Gateway :8080 → [redirecciona según la URL] → Microservicio destino

Cada microservicio se registra en Eureka :8761 al iniciar, para que el Gateway pueda encontrarlo.
Nota: Todos los microservicios comparten el mismo patrón de arquitectura interna: Controller → Service → Repository (Controlador recibe la petición, Service aplica la lógica de negocio, Repository accede a la base de datos).

2. Servicios de Infraestructura

2.1 Eureka Server — Servidor de Descubrimiento

Puerto: 8761

Es el directorio central del sistema. Cada microservicio al iniciar se registra aquí con su nombre y dirección IP. El API Gateway consulta Eureka para saber dónde está cada servicio antes de redirigir una petición. Sin Eureka, los servicios no pueden encontrarse entre sí.

  • Consola web de administración: http://localhost:8761
  • Muestra qué servicios están activos en tiempo real

2.2 API Gateway — Puerta de Entrada

Puerto: 8080

Es el único punto de entrada del sistema. Todos los clientes (Postman, aplicación web, app móvil) siempre envían sus peticiones al Gateway. Este analiza la URL y la redirige al microservicio correspondiente usando la lista de rutas configuradas.

3. Microservicios de Negocio

3.1 Auth Service — Autenticación y Seguridad

Puerto: 8081   Base de datos: auth_db

Es el microservicio encargado de la seguridad del sistema. Gestiona el registro de usuarios, el inicio de sesión y la generación de tokens JWT (JSON Web Token). Cuando un usuario inicia sesión, recibe un token que identifica quién es y qué rol tiene dentro del sistema.

¿Qué hace?

  • Registrar nuevos usuarios con username, email y contraseña (encriptada con BCrypt)
  • Validar credenciales al hacer login y generar un token JWT con duración de 24 horas
  • Verificar si un token JWT es válido (usado por el Gateway)

Roles disponibles: USER, ADMIN, PROPIETARIO

MétodoEndpointDescripción
POST/api/v1/auth/registerRegistrar nuevo usuario
POST/api/v1/auth/loginIniciar sesión y obtener token JWT
GET/api/v1/auth/validate?token=...Validar si un token es válido
GET/api/v1/auth/healthVerificar estado del servicio

3.2 Usuario Service — Gestión de Perfiles

Puerto: 8082   Base de datos: usuario_db

Almacena el perfil completo de los usuarios del sistema: su nombre, apellido, email, teléfono y ciudad. Es diferente al auth-service: mientras auth-service guarda las credenciales de acceso (username y contraseña), usuario-service guarda los datos personales del usuario.

¿Qué hace?

  • Crear y actualizar perfiles de usuarios
  • Buscar usuarios por ID, email o ciudad
  • Borrado lógico: al "eliminar" un usuario, solo se marca como inactivo (activo=false), el registro no se borra

3.3 Propiedad Service — Gestión de Propiedades

Puerto: 8086   Base de datos: propiedad_db

Es el núcleo del sistema inmobiliario. Gestiona todas las propiedades disponibles para arriendo. Solo maneja dos tipos de propiedad: casas y departamentos. Cada propiedad tiene un ciclo de vida con tres estados posibles.

Tipos: CASA, DEPARTAMENTO

Estados: DISPONIBLE → puede reservarse | ARRENDADA → ocupada | INACTIVA → desactivada

¿Qué hace?

  • Publicar nuevas propiedades con sus características (habitaciones, metros cuadrados, precio, ubicación)
  • Buscar propiedades con filtros: tipo, ciudad, región, comuna, precio mínimo y máximo
  • Cambiar el estado de una propiedad (usado por reserva-service automáticamente)
  • Borrado lógico: eliminar cambia el estado a INACTIVA, no borra el registro

GET/api/v1/propiedades/todasListar todas (incluye ARRENDADA e INACTIVA)

GET/api/v1/propiedades/buscar?tipo=&ciudad=&precioMax=Búsqueda avanzada con filtros
GET/api/v1/propiedades/region/{region}Filtrar por región
GET/api/v1/propiedades/comuna/{comuna}Filtrar por comuna
GET/api/v1/propiedades/propietario/{id}Propiedades de un propietario

PATCH/api/v1/propiedades/{id}/estado?estado=Cambiar estado
DELETE/api/v1/propiedades/{id}Desactivar propiedad

3.4 Reserva Service — Gestión de Arriendos

Puerto: 8083   Base de datos: reserva_db

Gestiona el proceso completo del arriendo de una propiedad. Cuando se crea una reserva, automáticamente verifica que la propiedad esté disponible y la marca como arrendada. Cuando se cancela o completa, la propiedad vuelve a estar disponible. Esta comunicación se realiza mediante llamadas HTTP internas a propiedad-service usando Feign.

Estados de la reserva: PENDIENTE → CONFIRMADA → COMPLETADA / CANCELADA

¿Qué hace?

  • Crear reservas validando que la propiedad esté en estado DISPONIBLE
  • Cambiar automáticamente el estado de la propiedad a ARRENDADA al crear una reserva
  • Liberar la propiedad (volver a DISPONIBLE) al cancelar o completar la reserva
  • Consultar reservas por usuario o por propiedad
Integración importante: Reserva-service llama a propiedad-service internamente via Feign (HTTP) para verificar el estado y cambiar el estado de la propiedad. Utiliza OkHttp como cliente HTTP para soportar el método PATCH.

GET/api/v1/reservas/{id}Buscar reserva por ID
GET/api/v1/reservas/usuario/{id}Reservas de un usuario
GET/api/v1/reservas/propiedad/{id}Reservas de una propiedad
POST/api/v1/reservasCrear reserva
PATCH/api/v1/reservas/{id}/estado?estado=Cambiar estado de la reserva
DELETE/api/v1/reservas/{id}Eliminar reserva

3.5 Reseña Service — Calificaciones de Propiedades

Puerto: 8084   Base de datos: resena_db

Permite a los usuarios calificar y comentar las propiedades que han arrendado. La calificación va de 1 a 5 estrellas. El servicio también calcula estadísticas como el promedio de calificaciones y la cantidad total de reseñas por propiedad. Verifica que la propiedad exista llamando a propiedad-service antes de crear una reseña.

¿Qué hace?

  • Crear reseñas con calificación (1-5) y comentario de hasta 1000 caracteres
  • Calcular el promedio de calificaciones de una propiedad
  • Consultar reseñas por propiedad o por usuario
  • Actualizar y eliminar reseñas propias

3.6 Búsqueda Service — Motor de Búsqueda

Puerto: 8089   Base de datos: ninguna propia

Es un servicio especial que no tiene base de datos propia. Actúa como un intermediario inteligente que consulta los datos de propiedad-service y aplica filtros adicionales que ese servicio no soporta, como filtrar por número mínimo de habitaciones o metros cuadrados mínimos. Está diseñado para alimentar el buscador principal de la aplicación.

¿Qué hace?

  • Buscar propiedades disponibles con múltiples filtros: tipo, ciudad, región, comuna, precio, habitaciones, metros cuadrados
  • Retornar todas las propiedades disponibles sin filtros
  • Retornar las 10 propiedades más recientes como "propiedades destacadas" para la portada
MétodoEndpointDescripción
GET/api/v1/busqueda?tipo=&ciudad=&precioMax=&habitacionesMin=Búsqueda avanzada con filtros
GET/api/v1/busqueda/disponiblesTodas las propiedades disponibles
GET/api/v1/busqueda/destacadasTop 10 propiedades destacadas

3.7 Notificación Service — Sistema de Avisos

Puerto: 8087   Base de datos: notificacion_db

Gestiona las notificaciones internas del sistema hacia los usuarios. Permite enviar avisos sobre reservas, reseñas, pagos y mensajes del sistema. Lleva el control de qué notificaciones han sido leídas y cuáles están pendientes, mostrando un contador de no leídas.

Tipos de notificación: RESERVA, RESENA, SISTEMA, PAGO, BIENVENIDA

¿Qué hace?

  • Crear notificaciones para usuarios específicos
  • Listar notificaciones de un usuario (todas o solo no leídas)
  • Marcar una o todas las notificaciones como leídas
  • Mostrar el resumen con el conteo de notificaciones no leídas

3.8 Imagen Service — Gestión de Fotografías

Puerto: 8088   Base de datos: imagen_db

Administra las imágenes asociadas a cada propiedad. Almacena la URL de la imagen, su tipo de archivo y tamaño. Gestiona cuál imagen es la "principal" de cada propiedad, garantizando que solo una imagen pueda ser la principal a la vez. Cuando se cambia la imagen principal, automáticamente desactiva la anterior.

¿Qué hace?

  • Registrar imágenes de propiedades con su URL, tipo MIME y tamaño en bytes
  • Gestionar la imagen principal de cada propiedad (solo puede haber una)
  • Listar imágenes de una propiedad ordenadas: principal primero, luego por fecha
  • Eliminar imágenes individualmente o todas las de una propiedad

3.9 Reporte Service — Generación de Reportes

Puerto: 8085   Base de datos: reporte_db

Genera reportes estadísticos del sistema consultando datos de otros microservicios. Llama a propiedad-service para obtener datos de propiedades y a reserva-service para datos de reservas. Los reportes generados se guardan en la base de datos para mantener un historial y poder consultarlos posteriormente.

Tipos de reporte: PROPIEDADES, RESERVAS, INGRESOS, GENERAL

¿Qué hace?

  • Generar reporte de propiedades: estadísticas por estado, tipo y ciudad
  • Generar reporte de reservas: estadísticas por estado y cálculo de ingresos totales de reservas completadas
  • Generar reporte general: combina datos de propiedades y reservas
  • Consultar el historial de reportes generados (del más reciente al más antiguo)

4. Comunicación entre Microservicios

Algunos microservicios necesitan datos de otros para funcionar. Esta comunicación se realiza mediante OpenFeign, una librería que permite llamar a otros servicios HTTP como si fueran métodos Java normales.

Servicio origen Llama a Para qué
reserva-service propiedad-service Verificar que la propiedad esté DISPONIBLE y cambiar su estado
resena-service propiedad-service Verificar que la propiedad exista antes de crear una reseña
busqueda-service propiedad-service Obtener propiedades y aplicar filtros adicionales
reporte-service propiedad-service Obtener datos de propiedades para generar reportes
reporte-service reserva-service Obtener datos de reservas para generar reportes

5. Manejo de Errores

Todos los microservicios implementan un GlobalExceptionHandler que captura las excepciones y devuelve respuestas JSON estructuradas con el mismo formato en todos los servicios.

Código HTTP Código interno Cuándo ocurre
200 OK—Operación exitosa
201 Created—Recurso creado exitosamente
400 Bad RequestVALIDATION_ERRORCampos inválidos o faltantes (@Valid)
400 Bad RequestBAD_REQUESTRegla de negocio violada (ej: propiedad no disponible)
404 Not FoundNOT_FOUNDRecurso no encontrado (ej: propiedad con id 9999)
500 Internal Server ErrorSERVER_ERRORError inesperado del servidor

Formato de respuesta de error (siempre el mismo en todos los servicios):

{
  "codigo": "NOT_FOUND",
  "mensaje": "Propiedad no encontrada con id: 9999",
  "timestamp": "2026-05-25T03:00:00.000"
}
